# Deployment

The Vectreal Platform is deployed on **Fly.io** (region `fra`), fronted by **Cloudflare** for DNS, edge caching, and Turnstile bot protection. Deployments are automated through GitHub Actions.

> The canonical source of truth for infrastructure configuration is [`terraform/README.md`](https://github.com/Vectreal/vectreal-platform/blob/main/terraform/README.md). This page provides a high-level overview and links to the detailed Terraform docs rather than duplicating them.

---

## Architecture

```
GitHub Actions CI/CD
       │
       ▼
Docker image → GHCR → Fly.io registry
       │
       ▼
Fly.io apps (region fra)
  ├── vectreal-platform          (production, vectreal.com)
  └── vectreal-platform-staging  (staging, staging.vectreal.com)
       │
       ▼
Cloudflare (DNS, edge cache, Turnstile)
       │
       ▼
Supabase (Auth + Postgres + Storage)
```

Terraform manages only the Cloudflare configuration (DNS records, Turnstile widgets, cache rules). Fly.io app configuration lives in `apps/vectreal-platform/fly.toml` (production) and `fly.staging.toml` (staging). Application data and 3D assets are hosted by Supabase.

## First-time setup

### Prerequisites

- A [Fly.io](https://fly.io) account with the `flyctl` CLI authenticated
- A Cloudflare account managing the `vectreal.com` zone
- Terraform ≥ 1.0
- A populated `.env.development` at the repo root

### Cloudflare infrastructure

```bash
cd terraform
./scripts/apply-infrastructure.sh   # terraform init + validate + apply
```

This provisions the Cloudflare DNS records, Turnstile widgets, and cache rules described in the [Terraform README](https://github.com/Vectreal/vectreal-platform/blob/main/terraform/README.md).

### Fly.io app secrets

Runtime secrets are synced to the Fly.io apps from your local `.env.development`:

```bash
# Staging
pnpm nx run terraform:setup-fly-secrets-staging

# Production
pnpm nx run terraform:setup-fly-secrets-prod

# Read-only check of current state
pnpm nx run terraform:verify-fly-secrets
```

## CI/CD workflows

Workflows live in `.github/workflows/`.

| Workflow | Fly app | Trigger | What it does |
|---|---|---|---|
| **Staging deploy** (`cd-platform-staging.yaml`) | `vectreal-platform-staging` | Push to `main` (on app/package changes) | Builds the Docker image, pushes to GHCR and the Fly.io registry, runs `flyctl deploy` against `fly.staging.toml` |
| **Production deploy** (`cd-platform-production.yaml`) | `vectreal-platform` | Manual `workflow_dispatch` | Runs the CI quality gate, then deploys the image against `fly.toml` |
| **Package release** | n/a | Managed by Release Please | Publishes `@vctrl/*` packages to npmjs.com |

Both deploy workflows call the shared `shared-fly-deploy.yaml` workflow. The build injects `VITE_PUBLIC_*` values as Docker build args; runtime secrets (Supabase, Stripe, Cloudflare Turnstile, email) are set as Fly.io app secrets.

### Required GitHub secrets

| Secret | Description |
|---|---|
| `FLY_API_TOKEN` | Fly.io deploy token |
| `VITE_PUBLIC_POSTHOG_TOKEN` / `VITE_PUBLIC_POSTHOG_HOST` | PostHog analytics config (build-time) |
| `VITE_PUBLIC_POSTHOG_UI_HOST` | Optional PostHog UI host |
| `VITE_PUBLIC_VECTREAL_API_KEY_PROD` | Optional public API key for demo scenes |
| `VITE_PUBLIC_DEMO_SCENE_URL` | Optional demo scene URL |

Application runtime secrets (Supabase URLs/keys, `DATABASE_URL`, `CSRF_SECRET`, Stripe keys, Cloudflare Turnstile keys) are managed on Fly.io via the `setup-fly-secrets` scripts rather than as GitHub secrets.

## Database migrations

Migrations follow a Supabase-first flow - Drizzle generates SQL from schema edits, the Supabase CLI applies it.

```bash
# Generate SQL from schema changes
pnpm nx run vectreal-platform:drizzle-generate

# Apply to staging
pnpm nx run vectreal-platform:supabase-db-push-staging

# Apply to production
pnpm nx run vectreal-platform:supabase-db-push-prod
```

Never hand-author migration SQL; generate it from schema modules in `app/db/schema/`.

---

## Docker

The platform app is built from a multi-stage `Dockerfile` at `apps/vectreal-platform/Dockerfile`. CI builds this image once, pushes it to GHCR and the Fly.io registry, and deploys it with `flyctl`. The container exposes a `/health` endpoint that the deploy workflow polls after each release.

---

## Staging-first rollout

Before promoting any change to production:

1. Push to `main` → the staging deployment runs automatically.
2. Validate on `staging.vectreal.com` (auth flows, 3D pipeline, embed preview).
3. Run the production deploy workflow manually once staging is green.
